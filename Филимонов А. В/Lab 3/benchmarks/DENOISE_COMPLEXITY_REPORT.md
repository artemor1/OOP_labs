# Denoise complexity analysis (Lab 3)

## Что протестировано
Проверена зависимость вычислительной сложности `SignalHandler.DeNoise` от:
- `rank`
- `iterations`
- длины массива `N`

Исходная реализация:
- строит Hankel матрицу `rows = N/2`, `cols = N - rows + 1`
- делает SVD полной Hankel-матрицы
- восстанавливает матрицу по первым `rank` сингулярным компонентам
- выполняет diagonal averaging (`HankelToArray`)  

См. реализацию в `SignalParseAndTransformation.cs`.

## Ограничения среды
В контейнере отсутствует .NET runtime/compiler (`dotnet`, `mono`, `mcs`), поэтому прямой профилировки C#-метода выполнить нельзя.
Сделан воспроизводимый Python-бенчмарк структуры алгоритма + аналитическая оценка FLOPs:
- реальный замер времени для частей `Hankel` и `HankelToArray`
- оценка FLOPs для SVD и reconstruction-части по `rank`

Скрипт: `benchmarks/denoise_complexity.py`  
Результаты: `benchmarks/denoise_complexity_results.csv`

## Основные результаты

### 1) Влияние `iterations`
Линейное: все части цикла умножаются на `iterations` (и SVD, и Hankel, и reconstruction).

### 2) Влияние `rank`
Почти линейное, но вклад невелик по сравнению с SVD при разумных `rank`.

Пример (`N=2048`, `iterations=3`):
- `rank=1`: total ~`21.500 GFlops`
- `rank=100`: total ~`22.123 GFlops`

Рост ~`2.9%`, потому что SVD доминирует.

### 3) Влияние длины массива `N`
Главный фактор — из-за SVD сложность близка к кубической `O(N^3)` (поскольку Hankel почти квадратная).

Для `rank=3`, `iterations=3`:
- `N=256`: `0.043 GFlops`
- `N=512`: `0.338 GFlops`
- `N=1024`: `2.694 GFlops`
- `N=2048`: `21.513 GFlops`
- `N=4096`: `171.950 GFlops`

Увеличение `N` в 2 раза даёт примерно рост нагрузки в ~8 раз.

## Является ли проблема «слишком большая Hankel матрица»?
**Да, но в первую очередь косвенно:**
- Большая Hankel-матрица делает SVD очень дорогой (`O(N^3)`), это основной потребитель CPU.
- Само построение Hankel и диагональное усреднение — `O(N^2)` и заметно дешевле SVD по асимптотике.

Также есть рост памяти:
- размер одной Hankel-матрицы примерно `(N/2)*(N/2+1)*8 байт`
- при `N=4096` это ~`32 MB` только для одной матрицы (без учёта матриц SVD/U/VT/R)

## Практический вывод
CPU-нагрузка в `DeNoise` почти наверняка вызвана не столько операцией заполнения Hankel, сколько **SVD этой большой Hankel-матрицы**.
